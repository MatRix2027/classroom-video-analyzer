# 课堂视频教师授课能力评价工具：项目立项与推进机制

## 1. 项目定位

本项目定位为一个面向教学质检、教师成长和培训复盘的 Web 工具。用户上传课堂视频后，系统自动完成语音转写、教学事件识别、关键帧抽取、视觉观察、授课能力评分和报告生成，最终输出一份可供质检人员审核、教师反馈、团队复盘使用的课堂分析报告。

项目当前名称：`classroom-video-analyzer`

目标用户：
- 教学质量负责人：快速判断一节课是否达标，定位红线风险和改进重点。
- 教学质检员：用统一标准完成多维度评分，减少人工主观波动。
- 教师培训负责人：提取优秀片段、典型问题和可复盘证据。
- 教师本人：理解自身授课优势、短板和下一步训练方向。

## 2. 核心问题

人工看课存在三个主要痛点：
- 单节课通常 40-60 分钟，完整质检耗时高。
- 不同质检员对“好课”和“问题课”的判断口径不完全一致。
- 优秀片段、红线行为、互动质量等证据难以快速沉淀。

本工具要解决的问题不是“替代人最终裁决”，而是把看课过程中的信息抽取、证据定位、初步评分和报告撰写自动化，让人把精力放在复核、校准和指导上。

## 3. 已有逻辑梳理

当前项目已经形成了比较完整的技术主线：

1. 上传视频
   - 前端支持本地视频上传。
   - 已有分块上传机制，解决大文件经过 Cloudflare Tunnel 时可能触发 524 超时的问题。
   - 支持的视频格式包括 `mp4`、`mov`、`mkv`、`webm`、`flv`、`avi`，上限为 2GB。

2. 创建分析任务
   - FastAPI 后端创建任务，SQLite 记录任务状态、进度、文件路径、评分结果。
   - 后台线程运行分析流程，前端通过轮询展示进度。

3. 视频与音频处理
   - 使用 FFmpeg 读取视频信息。
   - 抽取音频文件，供 ASR 使用。
   - 按教学事件和固定时间间隔抽取关键帧。

4. ASR 转写
   - 使用腾讯云 ASR。
   - 目标输出带时间戳、说话人信息的课堂转写文本。

5. 文本语义分析
   - 使用 LLM 从转写文本中识别教学事件。
   - 事件类型包括环节切换、互动指令、学生应答、教师反馈、知识节点、节奏信号、红线行为。
   - 已有互动链重构逻辑，用于把提问、回答、反馈串成可评价的教学互动链。

6. 视觉观察与混合评分
   - 已支持视觉模型配置。
   - 关键帧可用于观察教师仪表教态、板书/课件、学生状态、课堂氛围等。
   - 当前评分思路是文本模型负责大部分维度，视觉模型补充或直接评价需要看画面的维度。

7. 评分体系
   - 配置文件中已有 L1-L3、L4-L6、L7-L9、QC-v4、QA-v3 等标准。
   - 目前主线倾向使用统一 QC-v4 标准，同时保留班型标准兼容。
   - 评分维度覆盖教学内容、教学方法、教学规范、课堂效果等类别。
   - 红线项包括严重伤害学生心理健康、不干预危险行为、干扰正常教学、严重教学错误、违反教学环境要求等。

8. Web 展示
   - 已有上传页、分析进度页、仪表盘、详细报告页、历史记录页、评价标准页。
   - 仪表盘包含视频播放、总分、等级、雷达图、分维度详情和证据点跳转。

## 4. 产品目标

### 4.1 MVP 目标

MVP 要先做到“能稳定完成一节课的教师授课能力评价”：
- 上传一节课堂视频。
- 自动生成转写、事件、关键帧、评分卡和报告。
- 输出总分、等级、维度分、证据、风险项和改进建议。
- 支持质检人员在 Web 端查看视频和证据点。
- 分析失败时能明确告诉用户失败原因，并保留任务状态。

### 4.2 质量目标

MVP 阶段的质量目标：
- 单节视频能跑通完整流程。
- 报告结构稳定，可直接用于人工复核。
- 评分维度和评价标准可解释。
- 关键证据能跳转到视频对应位置。
- 视觉模型不可用时，系统可降级为文本评分，不阻断整体流程。
- 关键帧、教学事件、视觉评分状态必须结构化展示，避免把证据不足的 AI 初评误认为最终评价。

### 4.3 不做范围

当前阶段暂不追求：
- 完全无人审核的最终绩效裁决。
- 多用户权限体系。
- 大规模并发队列。
- 自动剪辑高光短视频。
- 复杂 BI 趋势分析。

这些可以作为后续 V2/V3 能力。

## 5. 当前风险和缺口

根据当前代码与测试状态，第一阶段需要优先处理这些问题：

1. 后端测试未完全通过
   - 当前测试结果为 `79 passed, 24 failed`。
   - 失败集中在 ASR 结果解析、LLM 重试、配置合并、pipeline 初始化时 API key 校验。

2. 配置兼容性不足
   - `ConfigManager._merge_config` 对旧格式配置存在未初始化变量问题。
   - Pipeline 测试用 mock key 时仍被真实 API key 校验阻断。

3. ASR 降级解析不稳
   - 需要兼容列表格式、`text` 简单文本格式、腾讯云结构化结果格式。
   - 简单文本应按换行或标点拆分为多个 segment。

4. LLM 重试策略与测试预期不一致
   - 当前只尝试 1 次，测试期望失败后可以重试成功。
   - 需要把重试次数配置化，默认至少 2 次。

5. Web 产品形态还偏工具页
   - 上传页已有能力介绍，但还需要更像一个质检工作台。
   - 分析模式、评价标准、证据点、报告导出等需要统一成清晰工作流。

6. 报告可用性需要提升
   - 当前报告以 Markdown/HTML 为主。
   - 后续需要支持可下载 PDF 或 Word，并强化“给教师反馈”的表达结构。

## 6. 版本路线

### Phase 0：接管与稳定

目标：让项目可维护、可运行、可验证。

交付：
- 明确项目定位和推进机制。
- 修复测试中暴露的基础问题。
- 保证前端构建通过。
- 保证后端基础测试通过。
- 整理启动方式和本地验证清单。

验收标准：
- `npm.cmd run build` 通过。
- `python -m pytest -q --basetemp .pytest_tmp` 基础测试通过或剩余问题有明确解释。
- 上传、任务创建、状态轮询、报告查看流程可本地验证。

### Phase 1：教师授课能力评价 MVP

目标：形成可真实使用的单节课评价闭环。

交付：
- 上传页改为“授课能力评价”工作入口。
- 分析任务支持选择评价场景：质检评价、教师成长反馈、优秀课例提取。
- 仪表盘突出总分、等级、红线、四大类表现、核心证据。
- 详细报告按“总体结论、维度评价、证据片段、改进建议、复核备注”组织。
- 评价标准页能清楚展示 QC-v4 和各班型标准。

验收标准：
- 质检员能从上传到报告完成一次闭环。
- 报告中的每个低分维度至少有一句证据或原因。
- 红线项独立展示，不被总分掩盖。

### Phase 2：评价准确性与校准

目标：把评分从“可生成”提升到“可信赖”。

交付：
- 建立样本视频校准集。
- 每个样本保留人工评分、AI 评分、差异原因。
- Prompt 和评分标准迭代有版本记录。
- 引入人工复核字段，支持质检员修正 AI 结论。

验收标准：
- 至少 10 节样本课完成校准。
- 对每个维度形成常见误判清单。
- AI 评分与人工评分差异能被追踪和复盘。

### Phase 3：团队使用增强

目标：服务批量质检、教师成长和培训素材沉淀。

交付：
- 历史记录支持筛选教师、班型、等级、时间。
- 支持导出 PDF/Word。
- 支持优秀片段/问题片段标记。
- 支持同一教师多次课程横向对比。

验收标准：
- 管理者能用历史数据做周/月复盘。
- 培训负责人能快速找到可引用片段。

## 7. 工作推进机制

### 7.1 工作原则

- 先跑通，再变漂亮。
- 先稳定评价闭环，再扩展高级能力。
- 每次改动都要能被一个命令或一个页面验证。
- 评价标准、Prompt、代码逻辑分开管理，避免混在一起难以校准。
- AI 结论必须尽量带证据，不能只有分数。

### 7.2 任务拆解方式

每个开发任务按以下格式定义：

- 背景：为什么要做。
- 用户价值：谁会因此更容易完成什么工作。
- 范围：这次做什么，不做什么。
- 影响文件：前端、后端、配置、文档。
- 验收方式：命令、页面路径、样本数据或截图。
- 风险：可能影响的已有能力。

### 7.3 优先级规则

P0：
- 阻断上传、分析、报告查看的功能。
- 会导致评分明显错误或报告不可用的问题。
- 会导致测试无法验证核心流程的问题。

P1：
- 提升质检效率和报告可读性的功能。
- 提升评分解释性的功能。
- 提升前端工作台体验的功能。

P2：
- 批量处理、趋势分析、素材剪辑、权限、多用户等增强能力。

### 7.4 每轮迭代节奏

每轮迭代建议控制在 1-3 天内，保持小步可验证。

固定流程：
1. 选定本轮目标。
2. 更新任务清单。
3. 修改代码或配置。
4. 跑前端构建和后端测试。
5. 本地打开页面验证关键路径。
6. 记录完成项、遗留项和下一轮建议。

### 7.5 验证清单

基础验证：
- `npm.cmd run build`
- `.venv\Scripts\python.exe -m pytest -q --basetemp .pytest_tmp`
- 后端健康检查：`GET /api/health`
- 前端上传页可打开
- 历史记录页可打开
- 评价标准页可打开

业务验证：
- 上传小视频并创建任务。
- 状态从上传、分析到完成可见。
- 完成后能看到总分、等级、维度分。
- 证据点能跳转视频时间。
- 报告页能打开。

### 7.6 决策记录

重大产品或技术决策需要写入 `docs/DECISIONS.md`，格式如下：

```markdown
## YYYY-MM-DD 决策标题

- 背景：
- 选项：
- 决策：
- 原因：
- 影响：
- 后续复盘点：
```
